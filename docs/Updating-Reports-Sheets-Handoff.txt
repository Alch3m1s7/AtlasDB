



## 1. System Purpose

Its first real-world goal is to replace fragile Google Sheets workflows with reliable Python ingestion, audit logs, snapshot identity, and queryable marketplace data.

## 5. Data Flow

### Current validated flow

1. Trigger:

powershell
python main.py ingest-spapi


2. SP-API checks recent DONE reports.
3. If recent DONE report exists, system reuses it instead of creating a new report.
4. Downloads raw tab-delimited inventory report.
5. Parses and normalises rows.
6. Exports JSONL.
7. Inserts valid rows into PostgreSQL.
8. Generates one snapshot_id for that batch.
9. Prints latest-snapshot inventory summary.

## 6. Critical Design Decisions & WHY

### Use one PostgreSQL/Supabase database with multiple schemas

Chosen schemas:

text
reference
audit
staging
atlas_core
atlas
keepau


Why:

* Less complexity than three physical databases.
* Easier joins across ASIN, marketplace, inventory, FX, and history.
* Still gives logical separation.

### Rename atlas_catalog to atlas

Chosen because user prefers simpler naming.

### ASIN is product identity; SKU is listing identity

Why:

* SKUs are marketplace-specific.
* Multiple SKUs may map to the same ASIN within the same marketplace.
* ASINs are tracked across five marketplaces.

### Store original price/currency as truth

Why:

* Converted GBP value is derived/cache only.
* Fixed FX rates are intentionally used for stable business comparison.
* User does not need daily FX accuracy.

### Use fixed FX rates, not live FX rates

Why:

* User already uses fixed rates in Google Sheets.
* Rates are updated monthly/quarterly only when materially different.
* This simplifies cross-market comparison and aligns with existing business logic.

### KeepAU is highest strategic priority

Why:

* Amazon AU lacks Keepa data.
* Historical Buy Box/BSR collection is most valuable to user.
* Goal is simplified Keepa-like AU dataset.

### Avoid partitioning at MVP stage

Why:

* Current data volume is tiny.
* Partitioning adds operational complexity.
* Add later when history volume justifies it.

## 7. Constraints & System Realities

* SP-API Reports API is asynchronous.
* FATAL is terminal for a report ID.
* Direct report generation for GET_FBA_MYI_UNSUPPRESSED_INVENTORY_DATA may fail.
* Reusing recent DONE reports is more reliable than forcing new reports.
* Latest snapshot queries must filter by snapshot_id.
* Absence from a report does not mean marketplace unavailability.
* VA access should be read-only first.
* Claude Code is useful for implementation but should not replace deterministic Python logic.
* No scraping planned for Atlas; data sources are SP-API, Keepa XLSX, Analyzer.tools XLSX, SellerSnap/BQool.

## 8. Failure Modes & Handling Strategy

| Failure                                        | Current Handling                                | Needs Improvement                                      |
| ---------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------ |
| SP-API report returns FATAL                  | Stops gracefully; does not retry same report ID | Store richer failed run metadata                       |
| Duplicate source file                          | Skips insert                                    | Add clear SKIPPED_DUPLICATE status                   |
| Old pre-migration rows have NULL snapshot_id | Excluded from latest snapshot query             | Acceptable                                             |
| Query aggregates all historical rows           | Fixed                                           | Keep protected with tests                              |
| Temporary test-ingest exists                 | Should be removed                               | Remove before commit                                   |
| Missing FX rate                                | Not built yet                                   | Since FX is fixed, validate config/table before ingest |
| Bad XLSX import                                | Not built yet                                   | Use staging + validation + audit log                   |

## 11. Safe vs Dangerous Changes

### Safe

* Add new schemas/tables without dropping existing tables.
* Add read-only queries.
* Add indexes.
* Add audit/run metadata.
* Add fixed FX table/config.
* Add staging tables.
* Add marketplace reference table.

### Dangerous

* Removing cache-before-create SP-API logic.
* Treating FATAL as retryable on same report ID.
* Aggregating inventory without snapshot_id.
* Replacing UUID batch snapshot_id with BIGSERIAL.
* Treating SKU as product identity.
* Treating missing ASIN from report as unavailable.
* Giving VA direct write permissions too early.
* Starting 200k AU ASIN collection before proving SP-API limits/cost.
